## 1. 문제 정의

### 핵심 문제

개인 투자자는 어닝콜 발표 직후 수 분 내에 매매 결정을 내려야 하는 초고변동성 상황에 맞닥뜨린다.

| 문제 | 현재 상태 (Trading Terminal 없이) |
|------|----------------------------------|
| **정보 처리 속도** | 어닝콜 원문을 직접 듣거나 뒤늦은 뉴스로 확인. 시장은 이미 반응 완료 |
| **감정적 의사결정** | 급등/급락 중 패닉 셀링·뇌동매매 발생. 사후에 후회 반복 |
| **수동 주문 지연** | EarningWhisperer 웹앱에서 신호를 보고 증권사 앱을 별도로 열어 수동 주문. 수 초~수십 초 슬리피지 발생 |
| **API 키 보안 리스크** | 증권사 API 키를 클라우드 서버에 맡기는 것에 대한 불안감. 기존 자동매매 SaaS의 구조적 취약점 |
| **매매 전략 부재** | "얼마나 살까", "언제 팔까"에 대한 규칙이 없어 일관성 없는 매매 반복 |

### Trading Terminal이 해결하는 것

- AI 신호 수신 즉시 OS 레벨 알림 → **정보 지연 0**
- 사용자가 설정한 리스크 룰 기반 자동/반자동 주문 → **감정 배제**
- KIS API 키를 사용자 로컬 PC에만 저장 → **보안 리스크 제거**
- 체결 결과를 백엔드로 자동 보고 → **포트폴리오 일관성 유지**

## 2. 사용자 페르소나

### 페르소나 A — "빠른 재량적 트레이더" 이준호 (32세, IT 직장인)

- **배경:** 주식 투자 경력 5년. 미국 주식 비중 높음. 어닝 시즌마다 NVDA, META, MSFT 어닝콜을 직접 챙겨봄
- **현재 불편함:**
  - 어닝콜 중 영어를 100% 이해 못해 핵심 뉘앙스를 놓침
  - 신호를 보고 증권사 MTS를 열어 주문하는 동안 이미 2~3% 움직임
  - 야간(한국 시간 새벽 2~4시) 어닝콜을 졸린 상태로 대응
- **원하는 것:** "신호 뜨면 내가 OK 버튼 하나만 누르면 알아서 살 수 있으면 좋겠다"
- **사용 모드:** SEMI_AUTO (신호 수신 → 확인 후 1-click 실행)
- **플랜:** PRO

### 페르소나 B — "검증형 스마트 개미" 박수진 (28세, 대학원생)

- **배경:** 금융공학 전공. 유튜버/커뮤니티 말 신뢰 안 함. 데이터 기반 투자 지향
- **현재 불편함:**
  - 어닝콜 AI 분석 서비스가 있어도 실제 주문까지 연결되는 툴이 없음
  - 어닝 직후 자동매매 기능은 기관/개발자 전용 영역이라 진입장벽 높음
  - 매매 후 체결가·수익률 기록 관리가 번거로움
- **원하는 것:** "시그널이 얼마나 정확한지 데이터로 확인하고, 작은 금액으로 먼저 자동 실행 테스트해보고 싶다"
- **사용 모드:** AUTO_PILOT (소액으로 룰 기반 자동 실행)
- **플랜:** PRO

## 3. 핵심 기능 목록

### MVP 포함 기능

| 기능 | 설명 | 우선순위 |
|------|------|---------|
| **F01 — WebSocket 신호 수신** | 백엔드 STOMP Private Queue에서 매매 명령 수신 | P0 |
| **F02 — KIS API 키 로컬 관리** | OS Credential Manager에 API Key/Secret 저장·조회 | P0 |
| **F03 — 트레이딩 모드 전환** | MANUAL / SEMI_AUTO / AUTO_PILOT 모드 UI 및 전환 로직 | P0 |
| **F04 — 주문 실행 (KIS API)** | KIS 해외주식 주문 API 호출 (시장가) | P0 |
| **F05 — 체결 콜백 전송** | 체결 결과를 백엔드 `/api/v1/trades/{tradeId}/callback`으로 POST | P0 |
| **F06 — 포트폴리오 동기화** | KIS 잔고 조회 후 `/api/v1/portfolio/sync`으로 POST | P0 |
| **F07 — Fallback (자동 MANUAL 전환)** | WebSocket 연결 끊김 감지 → 즉시 MANUAL 전환 + OS 알림 | P0 |
| **F08 — 신호 확인 다이얼로그** | SEMI_AUTO 모드 전용 승인/거부 팝업 (5초 카운트다운) | P1 |
| **F09 — 실시간 신호 피드 UI** | 수신한 신호 이력, ticker, action, ema_score 표시 | P1 |
| **F10 — 리스크 룰 설정 UI** | 1회 매수 비율, 최대 보유 비중, 쿨다운, EMA 임계치 설정 | P1 |
| **F11 — 거래 내역 조회** | 체결된 주문 목록 (백엔드 `/api/v1/trades` 조회) | P1 |
| **F12 — 앱 인증 (JWT)** | 로그인 후 JWT 토큰 로컬 저장, WebSocket/REST 인증 | P0 |

### MVP 제외 기능 (v2 이후)

| 기능 | 제외 이유 |
|------|----------|
| 지정가 주문 | KIS API 복잡도 증가. 시장가로 MVP 충분 |
| 다중 계좌 지원 | 단일 KIS 계좌로 MVP 검증 후 확장 |
| 자체 백테스팅 뷰 | 백엔드 signal_history 데이터 활용 기능. 웹 UI에서 제공 예정 |
| 푸시 알림 (모바일) | 데스크톱 OS 알림으로 대체 |
| 다국어 지원 | 한국 사용자 타겟 MVP에서 불필요 |
| 토스증권/키움 API 연동 | KIS API 단일 연동으로 MVP 검증 후 확장 |

## 4. 사용자 스토리 & 인수 조건

---

### F01 — WebSocket 신호 수신

**사용자 스토리**  
Pro 유저인 나는, 어닝콜 중 백엔드가 보낸 매매 명령을 실시간으로 수신하고 싶다. 그래야 내가 설정한 모드에 따라 즉각적으로 대응할 수 있다.

**인수 조건**

- [ ] 앱 기동 시 백엔드 WebSocket에 STOMP 연결을 자동으로 시도한다
- [ ] STOMP CONNECT 프레임의 `Authorization` 헤더에 저장된 JWT를 포함한다
- [ ] `/user/{userId}/queue/signals` 구독에 성공하면 상태 표시줄에 "연결됨" 표시
- [ ] Contract 3b 형식의 메시지 수신 시 `trade_id`, `action`, `target_qty`, `ticker`, `ema_score`를 정상 파싱한다
- [ ] 파싱 실패(형식 오류, 필드 누락) 시 해당 메시지를 무시하고 에러 로그를 남긴다. 앱은 계속 동작한다

---

### F02 — KIS API 키 로컬 관리

**사용자 스토리**  
나는 내 증권사 API 키가 클라우드 서버에 올라가는 것이 불안하다. 내 PC에서만 관리되고 싶다.

**인수 조건**

- [ ] 최초 실행 시 API Key, API Secret 입력 UI를 제공한다
- [ ] 입력된 값은 OS Credential Manager에 저장된다 (Windows: `credential-manager`, macOS: `keychain`)
- [ ] 저장 후에는 입력 필드에 값을 표시하지 않는다 (마스킹)
- [ ] 앱 재시작 시 OS Credential Manager에서 자동으로 로드한다
- [ ] 키 변경(재입력) 및 삭제 기능을 제공한다
- [ ] API Key 미설정 상태에서 AUTO_PILOT 또는 SEMI_AUTO 실행 승인 시 에러 메시지와 함께 설정 화면으로 유도한다

---

### F03 — 트레이딩 모드 전환

**사용자 스토리**  
나는 상황에 따라 자동 실행 수준을 조절하고 싶다. 오늘은 직접 확인하고 싶고, 바쁜 날은 자동으로 돌리고 싶다.

**인수 조건**

- [ ] 현재 모드(MANUAL / SEMI_AUTO / AUTO_PILOT)를 항상 화면 상단에 명확하게 표시한다
- [ ] 모드 전환 즉시 서버에 `PUT /api/v1/users/settings`를 호출하여 동기화한다
- [ ] AUTO_PILOT 모드는 PRO 플랜 유저에게만 활성화된다. FREE 유저가 시도 시 업그레이드 안내를 표시한다
- [ ] AUTO_PILOT 활성화 전, "자동 주문이 실행됩니다. 계속하시겠습니까?" 확인 다이얼로그를 1회 표시한다
- [ ] 각 모드별 동작:
  - MANUAL: 신호 수신 시 OS 알림만 발송. 주문 실행 없음
  - SEMI_AUTO: 신호 수신 시 확인 다이얼로그 표시 → 사용자 승인 시 주문 실행
  - AUTO_PILOT: 신호 수신 즉시 자동 주문 실행

---

### F04 — 주문 실행 (KIS API)

**사용자 스토리**  
나는 신호가 확정되면 KIS 증권사를 통해 실제 주문이 자동으로 나가길 원한다. 직접 앱을 열 필요 없이.

**인수 조건**

- [ ] SEMI_AUTO 승인 또는 AUTO_PILOT 신호 수신 시 KIS 해외주식 시장가 주문 API를 호출한다
- [ ] 주문 직전 KIS 잔고 조회 API를 호출하여 실제 예수금을 확인한다
- [ ] 예수금 부족 시 `target_qty`를 하향 조정하여 가능한 수량으로 주문한다 (0주 이하가 되면 주문 취소)
- [ ] 주문 수량이 0으로 조정된 경우 "예수금 부족으로 주문 취소됨" 알림과 함께 백엔드에 `FAILED` 콜백을 전송한다
- [ ] 주문 API 호출은 `trade_id` 단위로 중복 실행을 방지한다 (동일 `trade_id` 2회 수신 시 2번째는 무시)

---

### F05 — 체결 콜백 전송

**사용자 스토리**  
주문이 실행된 후 그 결과가 백엔드에 정확히 반영되길 원한다. 내 포트폴리오 기록이 항상 정확해야 한다.

**인수 조건**

- [ ] 주문 실행 직후 KIS 체결 결과를 조회하여 백엔드 `POST /api/v1/trades/{tradeId}/callback`을 호출한다
- [ ] 요청 본문은 Contract 4 형식을 준수한다: `status`, `broker_order_id`, `executed_price`, `executed_qty`, `error_message`
- [ ] 체결 성공 시 `status: "EXECUTED"`, 실패 시 `status: "FAILED"`
- [ ] 콜백 전송 실패(네트워크 오류 등) 시 최대 3회 재시도 (1초, 3초, 5초 간격)
- [ ] 3회 재시도 실패 시 로컬에 실패 로그를 저장하고 "콜백 전송 실패" 경고를 표시한다

---

### F06 — 포트폴리오 동기화

**사용자 스토리**  
앱을 켤 때마다 내 실제 계좌 잔고가 백엔드와 자동으로 맞춰지길 원한다.

**인수 조건**

- [ ] 앱 기동 시 KIS 잔고 조회 API를 호출하여 `POST /api/v1/portfolio/sync`를 전송한다
- [ ] 매매 체결 콜백(F05) 완료 직후 동기화를 1회 추가로 실행한다
- [ ] 요청 본문은 Contract 4b 형식을 준수한다: `total_cash`, `holdings[{ticker, qty, avg_price}]`
- [ ] 동기화 성공 시 UI 하단에 "마지막 동기화: HH:MM:SS" 타임스탬프를 갱신한다
- [ ] 동기화 실패 시 에러 메시지를 표시하되 앱은 계속 동작한다 (비크리티컬 실패)

---

### F07 — Fallback (자동 MANUAL 전환)

**사용자 스토리**  
자동매매 중 연결이 끊기면 아무도 모르게 주문이 계속 나가는 사고가 생기면 안 된다. 즉시 안전 상태로 전환되어야 한다.

**인수 조건**

- [ ] WebSocket 연결이 끊긴 것을 감지하면 (STOMP DISCONNECT 또는 30초 이상 heartbeat 무응답) 즉시 트레이딩 모드를 MANUAL로 강제 전환한다
- [ ] 강제 전환 즉시 OS 네이티브 알림을 발송한다: "EarningWhisperer: 연결 끊김 — MANUAL 모드로 전환됨"
- [ ] 앱 내부에도 빨간색 배너로 "연결 끊김" 상태를 표시한다
- [ ] 재연결 시도는 5초 간격으로 최대 10회 수행한다
- [ ] 재연결 성공 시 이전 모드(SEMI_AUTO, AUTO_PILOT)로 자동 복원하지 않는다. 사용자가 직접 모드를 재선택해야 한다

---

### F08 — 신호 확인 다이얼로그 (SEMI_AUTO)

**사용자 스토리**  
반자동 모드에서 신호를 보고 내가 직접 승인해야 주문이 나가길 원한다. 하지만 클릭 한 번으로 빠르게 결정할 수 있어야 한다.

**인수 조건**

- [ ] SEMI_AUTO 모드에서 신호 수신 시 모달 다이얼로그를 표시한다
- [ ] 다이얼로그 내용: Ticker, Action(BUY/SELL), 예상 수량, EMA Score, 현재 모드
- [ ] 5초 카운트다운이 진행되며, 시간 초과 시 자동으로 "거부" 처리한다 (주문 미실행)
- [ ] "승인" 버튼 클릭 시 즉시 주문 실행 (F04) 흐름 진입
- [ ] "거부" 버튼 또는 시간 초과 시 해당 `trade_id`에 대해 `FAILED` 콜백을 전송한다 (`error_message: "USER_REJECTED"`)
- [ ] 다이얼로그가 열려 있는 동안 다음 신호가 도착하면 큐에 보관하고, 현재 다이얼로그 처리 완료 후 순차 표시한다

---

### F09 — 실시간 신호 피드 UI

**사용자 스토리**  
어떤 신호가 들어왔고 어떻게 처리됐는지 한눈에 보고 싶다.

**인수 조건**

- [ ] 신호 수신 시 피드에 카드 형태로 추가된다 (최근 50건 유지, 초과 시 오래된 것 삭제)
- [ ] 카드 표시 항목: 수신 시각, Ticker, Action(BUY/SELL), EMA Score, 처리 상태(PENDING/EXECUTED/FAILED/REJECTED)
- [ ] 처리 상태 색상 구분: EXECUTED(초록), FAILED/REJECTED(빨강), PENDING(노랑)
- [ ] 현재 WebSocket 연결 상태(연결됨/끊김/재연결 중)를 상단에 항상 표시한다

---

### F10 — 리스크 룰 설정 UI

**사용자 스토리**  
매매 한도와 쿨다운을 내가 직접 조절할 수 있어야 한다. 실수로 너무 크게 베팅하지 않도록.

**인수 조건**

- [ ] 설정 항목: 1회 매수 예수금 비율(0~100%), 종목 최대 보유 비중(0~100%), 쿨다운(분), EMA 임계치(0.0~1.0)
- [ ] 설정 변경 시 `PUT /api/v1/users/settings`를 호출하여 즉시 서버에 저장한다
- [ ] 저장 성공 시 "설정 저장됨" 토스트 메시지를 표시한다
- [ ] 앱 로드 시 `GET /api/v1/users/me`로 현재 설정값을 불러와 UI에 반영한다
- [ ] 각 항목에 입력 범위 유효성 검사를 수행하며, 범위 초과 시 저장 불가 처리한다

---

### F11 — 거래 내역 조회

**사용자 스토리**  
내가 이 앱을 통해 어떤 거래를 했는지 확인하고 싶다.

**인수 조건**

- [ ] `GET /api/v1/trades?page=0&size=20`으로 거래 내역을 페이징 조회한다
- [ ] 표시 항목: 체결 시각, Ticker, 매수/매도, 체결가, 체결 수량, 상태(EXECUTED/FAILED)
- [ ] 스크롤 하단 도달 시 다음 페이지를 자동 로드한다 (무한 스크롤)
- [ ] 데이터 로드 실패 시 "데이터를 불러올 수 없습니다. 다시 시도하세요" 메시지와 재시도 버튼을 표시한다

### F12 — 앱 인증 (JWT)

**사용자 스토리**  
한 번 로그인하면 다음에 앱을 켤 때 다시 로그인하지 않고 싶다.

**인수 조건**

- [ ] 최초 실행 시 이메일/비밀번호 로그인 화면을 표시한다
- [ ] 로그인 성공 시 JWT accessToken을 OS Credential Manager(또는 로컬 암호화 스토리지)에 저장한다
- [ ] 앱 재기동 시 저장된 토큰으로 자동 로그인을 시도한다
- [ ] 토큰 만료(401) 응답 수신 시 로그인 화면으로 이동하고 저장된 토큰을 삭제한다
- [ ] 로그아웃 기능을 제공하며, 로그아웃 시 토큰 및 KIS API 키를 모두 삭제한다

## 5. 비기능 요구사항

### 5.1 보안

| 항목 | 요구사항 |
|------|---------|
| **API 키 저장** | KIS API Key/Secret은 반드시 OS Credential Manager를 통해 암호화 저장. 평문 파일 저장 금지 |
| **JWT 저장** | OS Credential Manager 또는 Electron `safeStorage` API 사용. `localStorage` 저장 금지 |
| **통신 암호화** | 백엔드와의 모든 통신은 HTTPS/WSS 사용. HTTP 평문 통신 금지 (개발 환경 예외 허용) |
| **화면 잠금** | 앱이 백그라운드로 전환되어도 KIS API 키는 메모리 외 노출 금지 |
| **로그 보안** | 로컬 로그 파일에 KIS API Key/Secret, JWT 토큰 평문 기록 금지 |

### 5.2 성능

| 항목 | 요구사항 |
|------|---------|
| **신호 처리 지연** | WebSocket 메시지 수신 후 주문 API 호출 시작까지 500ms 이내 |
| **다이얼로그 표시** | SEMI_AUTO 신호 수신 후 확인 다이얼로그 렌더링까지 200ms 이내 |
| **OS 알림 발송** | Fallback 감지 후 OS 네이티브 알림 발송까지 1초 이내 |
| **앱 기동 시간** | 콜드 스타트(최초 실행 포함) 기준 3초 이내 메인 화면 표시 |
| **메모리 사용** | 상시 동작 중 메모리 사용 500MB 이하 유지 |

### 5.3 안정성 & 가용성

| 항목 | 요구사항 |
|------|---------|
| **WebSocket 재연결** | 연결 끊김 감지 후 5초 간격, 최대 10회 자동 재연결 시도 |
| **콜백 재시도** | 백엔드 콜백 전송 실패 시 1초/3초/5초 간격 3회 재시도 |
| **중복 주문 방지** | 동일 `trade_id`에 대한 주문은 앱 내부 상태 관리로 1회만 실행 보장 |
| **오프라인 처리** | 인터넷 연결 없는 상태에서 앱 기동 시 오프라인 상태 표시, MANUAL 모드 강제 유지 |
| **크래시 복구** | 비정상 종료 후 재기동 시 이전 세션 로컬 상태(미전송 콜백 등)를 복구 처리 |

## 6. 기술 리스크 & 의존성

### 리스크 1 — KIS API 안정성 (심각도: 높음)

- **내용:** KIS 해외주식 주문 API는 장 개시 시간대에 트래픽 집중으로 간헐적 타임아웃/오류 발생 이력이 있음. 어닝콜은 주로 장 마감 후(After-Hours)이나 동일 리스크 존재
- **완화 방안:**
  - 주문 API 타임아웃을 3초로 설정. 초과 시 즉시 `FAILED` 처리 및 콜백 전송
  - KIS API 에러 코드별 분기 처리 (잔고 부족, 거래 정지 등)
  - 실패 시 OS 알림으로 사용자에게 즉시 고지

### 리스크 2 — OS Credential Manager 플랫폼 차이 (심각도: 중간)

- **내용:** Windows `credential-manager`, macOS `keychain` API 동작 방식이 다름. Electron의 `safeStorage` API가 Linux에서 지원 불안정
- **완화 방안:**
  - `keytar` 라이브러리 사용으로 플랫폼 추상화
  - MVP는 Windows/macOS만 지원. Linux는 v2에서 검토
  - E2E 테스트를 Windows/macOS 각각 실행하여 검증

### 리스크 3 — STOMP WebSocket 연결 복잡도 (심각도: 중간)

- **내용:** STOMP over WebSocket은 순수 WebSocket 대비 연결 상태 관리(CONNECT/SUBSCRIBE/DISCONNECT), heartbeat, session 만료 처리가 복잡
- **완화 방안:**
  - `@stomp/stompjs` 라이브러리 활용
  - 연결 상태를 앱 전역 상태(Zustand)로 관리하여 모든 컴포넌트에서 일관되게 참조
  - Heartbeat 주기: 서버 10초, 클라이언트 10초 설정

### 리스크 4 — 중복 주문 (심각도: 높음)

- **내용:** 네트워크 재전송, 사용자 실수, 앱 재기동 등 다양한 원인으로 동일 `trade_id`에 중복 주문이 발생할 수 있음
- **완화 방안:**
  - 앱 내부에 처리된 `trade_id` Set을 메모리 + 로컬 파일로 이중 관리
  - 앱 재기동 시 로컬 파일에서 처리 완료된 `trade_id` 목록 복원
  - 백엔드도 `trade_id` 기준 멱등성 처리를 수행하나, 클라이언트 1차 방어가 필수

### 리스크 5 — 백엔드 배포 의존성 (심각도: 낮음)

- **내용:** Trading Terminal은 백엔드의 JWT 인증, WebSocket STOMP, REST API 전체에 의존. 백엔드 변경 시 Terminal이 영향받음
- **완화 방안:**
  - Contract 문서(api-spec.md)를 단일 진실 공급원으로 유지
  - 백엔드 API 변경 시 Contract 버전 명시 및 Terminal 팀 동시 공지
  - 로컬 개발 환경에서 백엔드 Mock 서버 제공 (STOMP + REST)

## 7. MVP 범위 정의

### MVP 목표

"Pro 사용자가 Trading Terminal을 설치하고, KIS API 키를 등록한 뒤, 어닝콜 신호를 받아 SEMI_AUTO 또는 AUTO_PILOT 모드로 실제 주문을 실행하고, 그 결과가 백엔드에 정확히 반영된다"

### MVP 포함 (12개 기능)

| 기능 | 설명 |
|------|------|
| F01 — WebSocket 신호 수신 | STOMP Private Queue 연결 및 신호 파싱 |
| F02 — KIS API 키 로컬 관리 | OS Credential Manager 저장/조회/삭제 |
| F03 — 트레이딩 모드 전환 | MANUAL / SEMI_AUTO / AUTO_PILOT |
| F04 — 주문 실행 (KIS API) | 시장가 주문 실행, 잔고 보정 |
| F05 — 체결 콜백 전송 | Contract 4 형식 POST |
| F06 — 포트폴리오 동기화 | 기동 시 + 매매 후 Contract 4b 형식 POST |
| F07 — Fallback | 연결 끊김 → MANUAL 강제 전환 + OS 알림 |
| F08 — 확인 다이얼로그 | SEMI_AUTO 5초 카운트다운 승인/거부 |
| F09 — 신호 피드 UI | 수신 신호 이력 카드 뷰 |
| F10 — 리스크 룰 설정 | 설정값 서버 저장/로드 |
| F11 — 거래 내역 조회 | 페이징 목록 |
| F12 — 앱 인증 (JWT) | 로그인/자동 로그인/로그아웃 |

### MVP 성공 기준 (OKR)

| 지표 | 목표값 |
|------|--------|
| 신호 수신 → 주문 실행 지연 (P95) | 500ms 이하 |
| 체결 콜백 전송 성공률 | 99% 이상 |
| Fallback 동작 정확도 | 연결 끊김 감지 후 100% MANUAL 전환 |
| 중복 주문 발생 건수 | 0건 |

### MVP 제외 항목 정리

| 항목 | 제외 근거 |
|------|----------|
| 지정가 주문 | KIS API 복잡도 증가 대비 MVP 가치 낮음 |
| 다중 증권사 연동 | KIS 단일 검증 후 확장 |
| 자동 이익실현/손절 | 백엔드 룰 엔진 확장 필요. v2 로드맵 |
| 백테스팅 리포트 UI | 웹 프론트엔드에서 제공 예정 |
| 차트 표시 | 신호 피드 텍스트 뷰로 MVP 대체 |

### 개발 순서 권고

```
1단계 (기반): F12(인증) → F01(WebSocket) → F02(KIS 키 관리)
2단계 (핵심): F04(주문 실행) → F05(콜백) → F06(동기화) → F07(Fallback)
3단계 (UX): F03(모드 전환) → F08(다이얼로그) → F09(신호 피드) → F10(설정) → F11(거래 내역)
```
