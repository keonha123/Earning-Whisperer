# ⚙️ Backend (매매 및 비즈니스 서버) 요구사항 정의서

## 1. 모듈의 역할 및 목표
이 모듈은 EarningWhisperer 프로젝트의 '심장'이자 '손' 역할을 담당합니다. 
AI 엔진이 판단한 매매 시그널을 실시간으로 수신하여, 유저의 계좌 잔고와 리스크 관리 룰(Rule)에 따라 최적의 매수/매도 수량을 계산합니다. 이후 실제 증권사 API를 호출하여 주문을 체결하고, 모든 상황을 프론트엔드 대시보드에 실시간으로 중계하는 역할을 수행합니다.

## 2. 핵심 기능 요구사항 (Core Features)

### [Feature 1] AI 매매 시그널 실시간 수신 및 필터링
- Redis의 `trading-signals` 채널을 구독(Subscribe)하여 AI 서버가 발행하는 JSON 시그널을 비동기적으로 수신합니다.
- **핵심 로직:** 짧은 시간에 쏟아지는 대량의 시그널 이벤트 스트림을 안전하게 필터링하고 병렬로 처리하여 지연 시간(Latency)을 최소화해야 합니다. (Java Stream API 적극 활용)

### [Feature 2] 포트폴리오 룰 엔진 및 리스크 관리
- 무조건 AI의 신호를 따르는 것이 아니라, 사용자의 현재 계좌 잔고(예수금)와 기보유 종목 비중을 조회하여 '실제 주문 가능한 수량'을 계산합니다.
- **제약사항:** AI 시그널의 `score`가 일정 기준치(예: 0.7) 미만이거나, 이미 해당 종목의 목표 비중을 채웠다면 주문을 무시(Hold)하는 방어 로직이 필수적입니다.

### [Feature 3] 증권사 API 연동 및 매매 실행
- 계산된 주문 정보를 바탕으로 외부 증권사 API(한국투자증권 모의투자 등)에 매수/매도 주문 REST API 요청을 보냅니다.
- **제약사항:** 외부 API 통신 시 발생할 수 있는 타임아웃(Timeout)이나 호출 제한(Rate Limit)에 대비한 철저한 예외 처리(Try-Catch, Retry 로직)가 구현되어야 합니다.

### [Feature 4] 프론트엔드 실시간 중계 (WebSocket)
- STT 텍스트 스트림, AI 매매 시그널, 최종 주문 체결 내역 등을 프론트엔드(React/Next.js) 대시보드로 실시간 브로드캐스팅합니다.
- **제약사항:** 클라이언트가 새로고침을 하지 않아도 화면의 데이터와 텐션 미터기가 부드럽게 업데이트되도록 양방향 통신 채널을 안정적으로 유지해야 합니다.

## 3. 입출력 명세 (I/O Specification)
- **Input 1:** Redis Pub/Sub 메시지 (AI 시그널)
- **Input 2:** Frontend의 HTTP Request (사용자 설정, 잔고 조회 등)
- **Output 1:** 외부 증권사 API로의 HTTP POST Request (매매 주문)
- **Output 2:** Frontend로의 WebSocket(STOMP) 메시지 푸시

## 4. 기술 스택 (Java)
- `Java 17` 이상 / `Spring Boot 3.x`: 코어 프레임워크
- `Spring Data JPA` / `MySQL`: 영구 데이터(회원, 포트폴리오, 주문 로그) 저장
- `Spring Data Redis`: AI 시그널 수신 및 분산 캐싱
- `Spring WebSocket` (STOMP): 프론트엔드 실시간 통신
- `WebClient` 또는 `OpenFeign`: 외부 증권사 API 비동기 호출

## 5. 완료 기준 (Definition of Done - DoD)
이 모듈의 개발이 완료되었다고 평가받으려면 다음 테스트를 통과해야 합니다.
1. [ ] 로컬 환경에서 Redis 채널에 가짜 시그널 JSON을 던졌을 때, 백엔드가 즉시 이를 낚아채어 파싱 에러 없이 로그를 찍어내는가?
2. [ ] 주문 로직이 DB의 잔고 데이터를 정확히 읽어와 적절한 수량으로 증권사 모의투자 API 주문에 성공하는가?
3. [ ] 프론트엔드와 WebSocket이 연결된 상태에서 시그널 수신 시, 1초 이내에 프론트엔드로 알림 메시지가 도달하는가?
