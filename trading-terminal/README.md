# 🖥️ Trading Terminal (데스크톱 트레이딩 실행 엔진) 요구사항 정의서

## 1. 모듈의 역할 및 목표
이 모듈은 EarningWhisperer 시스템의 **'실행 엔진(Execution Engine)'**이자 **'개인용 보안 금고(Vault)'** 역할을 담당한다.
자본시장법(미등록 투자일임업 방지) 및 증권사 API 약관(키 중앙 집중화 금지)을 엄격히 준수하기 위해, 실제 매매 주문은 중앙 서버가 아닌 사용자의 로컬 PC(본 프로그램)에서 직접 실행된다. 즉, 백엔드가 내린 판단(Signal)을 수신하여 사용자의 증권사 계좌로 주문을 전송하는 '행동 대장' 역할을 수행한다. **제로 트러스트(Zero Trust) 보안 원칙**에 따라 사용자의 KIS API 키는 중앙 서버로 절대 전송되지 않으며, 오직 본 로컬 데스크톱 앱의 OS 보안 영역에만 암호화되어 저장된다.

## 2. 핵심 화면 및 기능 명세 (Core Pages)

본 데스크톱 에이전트는 단순한 백그라운드 실행기를 넘어, 전문적인 퀀트 트레이딩 터미널(HTS) 수준의 풍부한 UI/UX를 제공한다. SaaS 웹 대시보드의 시각적 요소를 포함하되, **실제 매매 실행 및 통제권**에 초점을 맞추어 구성된다.

### [Common UI] 전역 레이아웃 (Global Layout)
- **기능:** 앱 전역에서 일관된 네비게이션과 즉각적인 알림을 제공한다.
- **구성 요소:**
  - **Top Header (상단 바):** 티커(Ticker) 전역 검색창 및 글로벌 알림(🔔). (예: 어닝콜 시작, 백그라운드 자동매매 체결 시 OS 네이티브 알림과 함께 Toast 노출)
  - **Left Sidebar (좌측 메뉴바):** 대시보드 / 라이브 트레이딩 룸 / 매매 히스토리 / 보안 설정 등 주요 메뉴 네비게이션.
  - **Status Bar (하단 상태표시줄):** KIS API 연결 상태, 백엔드 WebSocket(STOMP) 연결 상태, 현재 활성화된 트레이딩 모드(Manual/Auto)를 실시간 뱃지로 표시.

### [Page 1] 인증 및 보안 금고 (Auth & Vault)
- **기능:** SaaS 계정 연동 및 증권사 API 키를 로컬에 안전하게 저장하는 화면.
- **구성 요소:**
  - **SaaS 로그인:** 중앙 서버와 통신하기 위한 JWT 토큰 발급.
  - **Vault (API 키 등록):** KIS 모의투자 앱 키, 시크릿, 계좌번호 입력 폼. 입력된 데이터는 OS의 자격 증명 관리자(Credential Manager)에 암호화되어 로컬에만 보관되며, 화면에는 마스킹 처리되어 표시된다.

### [Page 2] 메인 대시보드 (Main Dashboard)
- **기능:** 로컬 에이전트가 직접 KIS 증권사 API를 호출하여 불러온 '가장 정확한 실제 계좌 상태'를 보여주는 오버뷰.
- **구성 요소:**
  - **내 모의투자 요약:** KIS 잔고 API를 통해 불러온 총 자산, 보유 현금, 실시간 수익률. (이 데이터는 백엔드 장부 동기화를 위해 전송됨)
  - **오늘의 일정 (Watchlist):** 당일 어닝콜이 예정된 타겟 기업 리스트. 어닝콜이 시작된 종목에는 🔴 **LIVE** 배지가 점멸하며 라이브 룸으로의 진입을 유도한다.

### [Page 3] 🔴 라이브 트레이딩 룸 (Live Trading Room) - 🌟 핵심 화면
- **기능:** 백엔드로부터 WebSocket 실시간 데이터를 수신하여 시각화하고, 로컬에서 즉각적인 매매를 실행하는 데스크톱 앱의 메인 터미널.
- **구성 요소 (3분할 레이아웃):**
  - **[좌측] 차트 패널:** 해당 종목의 실시간 주가 틱 차트 및 호가창.
  - **[중앙] AI 실시간 분석 패널:** - STT 발언 텍스트 실시간 타이핑 효과.
    - AI 판단을 시각화한 **'Raw Score 텐션 미터기(게이지)'** 및 노이즈가 제거된 **'EMA 추세선(Trend Line)'**.
  - **[우측] 매매 실행 및 시그널 피드:**
    - 백엔드가 전송한 매매 근거(Rationale) 및 시그널 피드.
    - 하단에 즉시 주문 가능한 **[매수] / [매도] 실행 버튼** 배치. (클릭 시 KIS 주문 API 즉시 호출)

### [Page 4] 📜 체결 영수증 및 히스토리 (Execution History)
- **기능:** 중앙 서버의 시그널과 로컬 PC의 실제 체결 내역을 대조하는 교차 검증 화면.
- **구성 요소:**
  - 특정 시간에 AI가 보낸 시그널 내역과, 로컬 에이전트가 KIS 증권사로 요청한 HTTP 응답(성공/에러 코드, 실제 체결가, 체결 수량)을 나란히 리스트업하여 투명한 거래 기록을 제공한다.

### [Page 5] 시스템 설정 및 트레이딩 통제권 (Settings & Control)
- **기능:** 룰 엔진의 글로벌 기준을 세팅하고, 앱의 자동화 수준을 제어하는 화면.
- **구성 요소:**
  - **리스크 관리 UI:** 1회 매수 예수금 비율, 최대 보유 비중, 쿨다운 타임 설정. (설정값은 백엔드 룰 엔진으로 `PUT` 전송되어 동기화됨)
  - **🚀 트레이딩 모드 토글 (3단계 스위치):** 유저의 심리적 저항감을 낮추기 위한 핵심 장치.
    1. **수동 모드 (Manual):** 시그널 피드만 수신하며 자동매매는 전면 차단. 유저가 라이브 룸에서 직접 매매 버튼을 클릭해야 함.
    2. **반자동 모드 (1-Click):** AI 매매 신호 수신 시 화면 중앙에 승인 팝업 노출. 유저가 [승인]을 클릭하거나 단축키(Enter)를 입력할 때만 로컬에서 주문 API가 호출됨.
    3. **완전 자동 모드 (Auto-Pilot):** 프론트엔드 개입 없이, 수신된 시그널에 따라 Main Process(Node.js)가 백그라운드에서 즉시 KIS API를 자동 호출하여 주문을 체결함.

## 3. 핵심 백그라운드 로직 (Core Logic / Main Process)

### [기능 1] 백엔드 신호 수신 (WebSocket Listener)
- Spring Boot 중앙 서버와 웹소켓(STOMP) 채널(`/user/{userId}/queue/signals`)을 연결하여 백그라운드에서 상시 대기한다.
- 신호가 도착하면 즉시 OS 네이티브 알림(Notification)을 띄우고, 설정된 트레이딩 모드에 따라 다음 로직을 분기한다.

### [기능 2] KIS OpenAPI 통신 클라이언트 (Order Executor)
- **토큰 관리:** KIS API 호출을 위한 Access Token을 발급받고 로컬 메모리에 캐싱하며, 만료 전 선제적으로 갱신한다. (기존 백엔드의 `KisTokenManager` 로직 이식)
- **수량 보정 후 주문 실행:** 백엔드로부터 수신한 `target_qty`(장부 기준 목표 수량)를 참고하되, 주문 직전 실제 KIS 잔고 API를 호출하여 예수금/보유 수량을 확인하고 필요 시 수량을 하향 보정한다. 보정된 수량으로 매수/매도 HTTP POST 요청을 전송한다. (Rate Limit 및 Timeout 예외 처리 포함)

### [기능 3] 체결 결과 및 잔고 콜백 (Sync API)
- **결과 보고:** KIS 서버로부터 주문 응답(증권사 주문번호, 체결 상태)을 받으면, 실제 체결 수량(`executed_qty`)을 포함한 결과를 백엔드 콜백 API(`POST /api/v1/trades/{tradeId}/callback`)로 전송하여 중앙 DB의 거래 상태(`PENDING` ➔ `EXECUTED` / `FAILED`)를 업데이트한다. 백엔드는 이 `executed_qty`로 자체 장부(Ledger)의 `target_qty` 기반 추정값 오차를 교정한다.
- **장부 동기화:** 실제 증권사 계좌의 잔고 및 보유 종목 정보를 조회하여 백엔드(`POST /api/v1/portfolio/sync`)로 전송함으로써, 백엔드의 논리적 룰 엔진이 사용하는 '자체 추정 장부(Internal Ledger)'의 오차를 교정한다.

## 4. Electron 특화 아키텍처 (IPC 통신)
본 모듈은 철저한 보안과 성능을 위해 Electron의 프로세스 분리 아키텍처를 따른다.
- **Renderer Process (UI/React):** 화면을 그리고 사용자의 클릭 이벤트를 처리한다. 보안상 로컬 파일 시스템이나 증권사 API에 직접 접근할 수 없다.
- **Main Process (Node.js):** KIS API 키 복호화, HTTP 주문 통신, 백엔드 웹소켓 연결 등 민감하고 무거운 작업을 백그라운드에서 처리한다.
- **IPC (Inter-Process Communication):** Renderer에서 "반자동 승인 버튼 클릭됨" 이벤트를 IPC 채널을 통해 Main으로 전달하면, Main 프로세스가 안전하게 증권사 API를 호출하는 브릿지 패턴을 적용한다.

## 5. 기술 스택
- **Framework:** `Electron` + `React` (Vite 빌드)
- **Main Process (Backend):** `Node.js`, `axios` (REST HTTP 통신), `sockjs-client` & `@stomp/stompjs` (웹소켓 통신)
- **Renderer Process (Frontend):** `Tailwind CSS`, `Zustand` (트레이딩 모드 상태 관리)
- **Security:** `keytar` 또는 `electron-store` (OS 레벨의 안전한 Credential 암호화 저장)

## 6. 완료 기준 (Definition of Done - DoD)
이 모듈의 개발이 완료되었다고 평가받으려면 다음 테스트를 통과해야 한다.
1. [ ] **보안 저장 테스트:** 입력된 KIS API 키가 평문으로 저장되지 않고, OS의 자격 증명 관리자(Windows Credential Manager / macOS Keychain)를 통해 안전하게 암호화되어 보관되는가?
2. [ ] **IPC 및 주문 실행 레이턴시:** 완전 자동 모드(Auto-Pilot) 상태에서 백엔드의 신호 수신 시, Renderer와 Main 간의 통신을 거쳐 KIS 서버로 주문이 전송되기까지의 지연 시간(Latency)이 1초 이내로 수행되는가?
3. [ ] **상태 동기화 무결성:** 로컬 주문 실행 직후 KIS에서 받은 응답(성공/실패)과 최신 잔고 현황이 백엔드의 Callback API로 누락 없이 전송되어, 백엔드의 상태와 일치하는가?
